# Crusam Windows Release Procedure

This document defines the required steps for cutting a Windows release build
of Crusam, and explains why each step exists. Follow this procedure for
**every** release — skipping the clean step is the most common cause of a
shipped `.exe` reporting the wrong version.

## Single source of truth: `pubspec.yaml`

`pubspec.yaml`'s `version:` field is the **only** place a release version is
declared:

```yaml
version: 1.0.1+2
```

- The part before `+` (`1.0.1`) becomes the Windows **File Version** and
  **Product Version** (major.minor.patch, with build number `0` unless
  `--build-number` is passed).
- The part after `+` (`2`) becomes the build number component.

Do not introduce a second place that declares the app version (no separate
`version.txt`, no hardcoded version string/constant in Dart, no manually
edited `.rc` file). If a workflow ever seems to need one, that's a signal the
build isn't picking up `pubspec.yaml` correctly — fix the build, don't add a
second source.

### How this flows into the compiled `.exe`

1. `flutter build windows` reads `version:` from `pubspec.yaml`.
2. It generates `windows/flutter/ephemeral/generated_config.cmake`, which
   defines `FLUTTER_VERSION`, `FLUTTER_VERSION_MAJOR`, `FLUTTER_VERSION_MINOR`,
   `FLUTTER_VERSION_PATCH`, and `FLUTTER_VERSION_BUILD`.
3. `windows/runner/CMakeLists.txt` passes those as preprocessor definitions
   to the runner target.
4. `windows/runner/Runner.rc` uses them to fill in `FILEVERSION` and
   `PRODUCTVERSION` in the embedded `VS_VERSION_INFO` resource.
5. That resource is compiled into `crusam.exe` and is exactly what Windows
   Explorer shows under **Properties → Details → File version / Product
   version**.
6. At runtime, `PackageInfo.fromPlatform()` (used by `UpdateService`) reads
   that same embedded resource back out. As long as step 1–5 ran cleanly,
   the app's self-reported version and the `.exe`'s Explorer-visible version
   always agree.

## Required release steps

Run these **in order**, from the `crusam/` directory:

1. **Update the version in `pubspec.yaml`.**
   Bump `version:` to the new release version, e.g. `1.0.1+2`.

2. **`flutter clean`**
   Removes `build/` and `windows/flutter/ephemeral/`. This is not optional —
   see "Why clean builds are required" below.

3. **`flutter pub get`**
   Re-resolves dependencies and regenerates the ephemeral Windows config
   (`generated_config.cmake`) from the `pubspec.yaml` you just edited.

4. **`flutter build windows --release`**
   Produces `build/windows/x64/runner/Release/crusam.exe` (path may be
   `build/windows/runner/Release/` on older Flutter versions).

5. **Verify the built executable before packaging anything.**
   - In Windows Explorer, right-click `crusam.exe` → **Properties** →
     **Details** tab.
   - Confirm **File version** and **Product version** exactly match the
     `version:` value in `pubspec.yaml` (e.g. `pubspec.yaml: 1.0.1+2` →
     Explorer should show `1.0.1.0`).
   - If they don't match, do not package or publish this build. Re-run
     from step 2.

6. Only after verification passes: package the release output as
   `CruSam-Setup-<version>.exe` (e.g. `CruSam-Setup-1.0.1.exe`) and publish
   it as an asset on a new GitHub release tagged `v<version>` (e.g.
   `v1.0.1`). See "How the app discovers updates" below — the in-app
   updater finds this release automatically; there is no separate metadata
   file to update.

## Why clean builds are required on Windows

Flutter's Windows build is a CMake project under the hood, and CMake's
incremental build system decides what to recompile based on file
timestamps and its own cache — not based on "did `pubspec.yaml`'s version
field change". In practice this means:

- `generated_config.cmake` is regenerated by `flutter pub get` /
  `flutter build windows`, but an existing `build/windows/...` CMake cache
  from a previous build can still consider the runner target "up to date"
  and skip recompiling `Runner.rc` into a new `.res` file, even though the
  `FLUTTER_VERSION_*` preprocessor definitions changed.
- Because the version is baked into the binary at **resource-compile time**,
  not read at runtime from a file, a stale incremental build can silently
  ship an `.exe` whose embedded File Version / Product Version reflects an
  *older* `pubspec.yaml` value than the one currently in the repo — even
  though the Dart code itself rebuilt correctly.
- This is exactly the failure mode that made `installed_version.txt` seem
  necessary in the first place: the app *looked* like it was stuck on an
  old version after "updating", when actually the shipped `.exe`'s
  PE resource had never been refreshed.

Running `flutter clean` before every release removes the `build/` directory
and the `windows/flutter/ephemeral/` folder entirely, forcing CMake to
reconfigure and recompile from scratch. This guarantees the `FILEVERSION`/
`PRODUCTVERSION` resource in the shipped `.exe` reflects the current
`pubspec.yaml`, every time.

## How the app discovers updates (GitHub Releases)

Crusam no longer relies on a manually-maintained `latest.json` file for
update information. `UpdateService.checkForUpdate()` queries the GitHub
Releases REST API directly:

```
GET https://api.github.com/repos/CruciaTos/CruSam/releases/latest
```

(The owner/repo pair is declared once, in `lib/core/updater/version_constants.dart`
as `kGitHubRepoOwner` / `kGitHubRepoName`, and the endpoint URL is built from
those constants — don't hardcode `"CruciaTos/CruSam"` anywhere else.)

From that response, the app:

1. Reads `tag_name` (e.g. `v1.0.1`) and strips a single leading `v`/`V` to
   get the version string (`1.0.1`) used for the update-available check.
2. Scans `assets[]` for a file matching the installer naming convention:
   **`CruSam-Setup-<version>.exe`** (e.g. `CruSam-Setup-1.0.1.exe`). GitHub's
   auto-generated source `.zip`/`.tar.gz` are ignored by construction — the
   matcher requires a `.exe` extension.
3. Uses that asset's `browser_download_url` as the update download URL.

**This means every release's uploaded asset must be named exactly
`CruSam-Setup-<version>.exe`, or the in-app updater will not find it** and
will silently report "no update information available" (see error handling
below) rather than erroring loudly. Double-check the asset filename before
publishing a release.

If GitHub is unreachable, no release has been published yet, the release
has no asset matching that convention, or the response can't be parsed,
`checkForUpdate()` returns a graceful "no update available" result instead
of throwing — the app simply won't prompt the user to update, rather than
showing an error.

### Known gap: installer vs. zip-based update pipeline

`UpdateService.downloadUpdate()` and `updater.exe` (`updater/bin/main.dart`)
still assume the downloaded file is a `.zip` that gets extracted in place —
that part of the pipeline was not changed by the move to GitHub Releases.
Since the installer asset matched above is a `.exe` (a real Windows
installer), running the existing zip-extraction updater against it will not
work correctly. Wiring the download/launch flow to run the `.exe` installer
directly (rather than extract a zip) is tracked as follow-up work and is
intentionally out of scope for this change, which is limited to *where the
app gets version and download-URL information from*.

## Can `latest.json` be deleted now?

**Not yet — wait one release cycle.**

As of this change, no code in this repository reads `latest.json` anymore;
`UpdateService` talks to the GitHub Releases API directly. In that narrow
sense it's dead weight in the repo today.

However, any Crusam install still running a **pre-Phase-4** build has the
*old* `UpdateService`, which checks `latest.json` (not GitHub Releases) to
learn about new versions. If `latest.json` is deleted immediately:

- Those older installs will get a 404 fetching it.
- They will silently stop detecting updates at all — including the update
  that would carry them onto this new GitHub-Releases-aware build.
- There's no way to push a fix to them once they can no longer see that a
  newer version exists.

So: keep `latest.json` in place, and keep publishing new releases the old
way alongside the new way (i.e. still updating it) for at least one release
cycle after the GitHub-Releases-aware build ships — long enough that every
install still capable of self-updating has had the chance to pick up the
new build and switch update-checking mechanisms on its own. After that,
`latest.json` and the code path that writes it as part of the release
process can be removed for good.

This mirrors the same reasoning as the `installed_version.txt` fallback
below: don't cut off an old mechanism until everything that depends on it
has had a chance to move off it.

## Note on `installed_version.txt`

`UpdateService.getCurrentVersion()` reads the running app's version primarily
via `PackageInfo.fromPlatform()`, which reflects the embedded PE resource
described above. `installed_version.txt` (written by `updater.exe` after an
in-place update) is kept only as a **temporary fallback** for compatibility
with installs that were updated by an older `updater.exe` build, and is only
consulted if `PackageInfo` ever reports an empty version — which should not
happen on a build produced via this procedure.

Do not extend or rely on `installed_version.txt` for anything new. It is not
a source of truth; it exists solely as a bridge for previously-updated
installs and is a candidate for removal in a future phase once no installs
depend on it.
